目前 app 里的处理方式是：复用同一个 LLM 后端服务 (llama-server)，通过 API 轮询调用。

具体实现如下：

1. 归纳阶段 (Indexing)：在解析文档（如处理图片或段落摘要）时，调用 
generate_summary
 方法，向本地 8080 端口发送 HTTP 请求。
2. RAG 生成阶段 (Chat)：在检索到相关内容后，调用 
generate_content
 方法，构造包含上下文的 Prompt，再次向同一个 8080 端口发送请求。

关于冲突： 底层的 llama-server 默认是串行处理请求的。这意味着如果你一边在解析超大文档（不断生成摘要），一边去提问，提问的请求会排队等待，直到当前的摘要任务完成。 对于个人桌面应用，这种策略能最大程度节省显存（因为不需要加载两个模型）

这种方案需要修改：
RAG 生成阶段 (Chat) 最高优先级，如果有对话请求，必须立即响应，不能等待
