# Knot RAG 架构局限性分析

> 基于 `docs/knot-markdown.md` 中记录的完整数据流，分析当前 RAG 系统在跨文档推理和文档内跨章节关联方面的不足。

---

## 一、当前架构概览

### 数据处理流程

```
文档 A ──→ 独立解析 ──→ 独立 embedding ──→ 独立入库（VectorRecord）
文档 B ──→ 独立解析 ──→ 独立 embedding ──→ 独立入库（VectorRecord）
文档 C ──→ 独立解析 ──→ 独立 embedding ──→ 独立入库（VectorRecord）
                                               │
                                          搜索时按相似度
                                          返回 N 个独立片段
                                               │
                                               ▼
                                          LLM 生成回答
```

### 每条 VectorRecord 知道什么

| 字段          | 信息           | 示例                    |
| ------------- | -------------- | ----------------------- |
| `text`        | 节点原始内容   | `"监督学习是一种..."`   |
| `file_path`   | 所属文件       | `"/docs/ml.md"`         |
| `breadcrumbs` | 文档内层级路径 | `"机器学习 > 监督学习"` |
| `parent_id`   | 父节点 ID      | `"ml.md-1"`             |
| `vector`      | 语义向量       | `[0.12, -0.34, ...]`    |

### 每条 VectorRecord 不知道什么

| 缺失信息       | 说明                                                                      |
| -------------- | ------------------------------------------------------------------------- |
| **兄弟节点**   | 不知道同级章节有哪些（如 "1.1 监督学习" 不知道 "1.2 无监督学习" 的存在）  |
| **文档全局**   | 不知道整个文档的主题、结论和摘要                                          |
| **跨文档关联** | 不知道其他文档是否有相关内容                                              |
| **引用关系**   | 不知道文档内或文档间的交叉引用（如 "参见第3章" 或 `[详见报告B](./B.md)`） |
| **时序关系**   | 不知道文档的创建/修改时间先后                                             |

## 二、问题分析

### 问题 1：跨文档推理能力缺失

**典型用户问题**：

- *"文档A的方案和文档B的方案有什么区别？"*
- *"这3份AI报告的共同趋势是什么？"*
- *"哪份报告对隐私保护的讨论最深入？"*

**当前系统行为**：

```
用户: "比较文档A和文档B的AI治理方案"

搜索过程:
  query embedding → 向量近邻搜索 → 按距离排序
  
可能的结果:
  ✅ 命中文档A的 "AI治理" 相关片段（score=0.82）
  ✅ 命中文档B的 "AI治理" 相关片段（score=0.78）
  ❌ 但也可能只命中一个文档（另一个距离太远被过滤）
  ❌ 两个片段之间没有 "对比" 关系的建模
  ❌ 没有文档级的全局摘要可供参考
```

**根因分析**：

1. **索引粒度过细**：每个 VectorRecord 是章节级片段，没有文档级的摘要/表示
2. **搜索是单轮的**：一次查询只做一次检索，无法根据第一轮结果发起补充检索
3. **无文档关联建模**：文档之间的引用链接（`[text](url)`）虽然被保留在 content 中，但没有被提取为结构化的关联关系

### 问题 2：同文档跨章节推理能力不足

**典型用户问题**：

- *"第3章的实验结果是否验证了第1章提出的假设？"*
- *"总结这篇文档的核心论点和支撑论据"*
- *"作者在结论部分是否回应了引言中提出的所有问题？"*

**当前系统行为**：

```
用户: "实验结果是否验证了假设"

搜索过程:
  query embedding → 可能命中 "第3章 > 实验结果" 的片段

问题:
  ❌ 不会自动关联 "第1章 > 研究假设" 的内容
  ❌ breadcrumbs 只告诉你 "第3章 > 3.1 实验结果"，不包含第1章的信息
  ❌ parent_id 存在但搜索时完全未利用
  ❌ "如第1章所述" 这种文内引用没有被索引
```

**根因分析**：

1. **`parent_id` 未被利用**：VectorRecord 中有 `parent_id` 字段，可以追溯到父节点，但搜索逻辑中从未使用它来扩展上下文
2. **无兄弟节点发现机制**：知道 "1.1 监督学习" 的父是 "第1章"，但无法高效查找 "1.2 无监督学习"
3. **Tree thinning 只管合并**：将碎片合并到父节点，但不建立章节间的语义联系
4. **Embedding 上下文有限**：`enrich_node()` 中的 breadcrumbs 只提供了向上的路径，没有提供同级或全局上下文

### 问题 3：缺少文档级表示

**典型用户问题**：

- *"这篇报告的核心观点是什么？"*
- *"这份PDF主要讲了哪几个方面？"*

**当前系统行为**：

```
搜索命中某些章节片段，但：
  ❌ 没有一条记录代表 "整个文档" 的摘要
  ❌ root 节点虽然存在，但 content 通常为空（只有子节点有内容）
  ❌ 用户需要看到的是全局概述，而不是某个章节的细节
```

## 三、对比：当前架构 vs 改进目标

### 当前能力矩阵

| 查询类型         | 当前能力   | 原因                                          |
| ---------------- | ---------- | --------------------------------------------- |
| 单点事实查找     | ✅ 优秀     | 向量/关键词搜索准确命中                       |
| 单文档单章节问答 | ✅ 优秀     | breadcrumbs 提供了准确的上下文定位            |
| 单文档跨章节推理 | ⚠️ 部分支持 | 依赖 LLM 自行从多个片段推理，但片段选择不可控 |
| 跨文档比较/对比  | ❌ 较弱     | 无法保证召回两个文档的相关片段                |
| 跨文档趋势总结   | ❌ 较弱     | 缺少文档级摘要，碎片化严重                    |
| 全文档摘要生成   | ❌ 不支持   | 没有文档级的索引记录                          |

---

## 四、改进方向

### 方案 1：上下文窗口扩展（推荐优先实施）

**核心思路**：当搜索命中某个节点时，自动拉取其 parent 和 sibling 节点作为补充上下文。

```
当前:
  搜索 → 返回 [片段X]
  
改进后:
  搜索 → 命中 片段X
       → 查询 parent_id → 获取父节点内容
       → 查询同一 parent_id 的其他节点 → 获取兄弟节点
       → 返回 [片段X + 父级上下文 + 相邻章节]
```

**改动范围**：
- 修改 `KnotStore.search()` 返回结果后的后处理逻辑
- 利用现有的 `parent_id` 和 `file_path` 字段
- 不需要修改索引结构

**复杂度**：⭐⭐

**收益**：解决同文档跨章节推理问题，LLM 能看到更完整的上下文

### 方案 2：文档级摘要索引

**核心思路**：在索引时为每个文档生成一条摘要记录，参与搜索。

```
当前:
  文档 → [章节1, 章节2, 章节3, ...]  （只有碎片）

改进后:
  文档 → [文档摘要, 章节1, 章节2, 章节3, ...]
          ↑ 新增：概括全文的摘要记录
```

**改动范围**：
- `KnotIndexer.index_file()` 完成后，对 root 节点生成 LLM 摘要
- 将摘要作为额外的 VectorRecord 入库
- 可以标记 `extra: { "type": "summary" }` 区分摘要与正文

**复杂度**：⭐⭐

**收益**：支持全文档摘要查询，改善跨文档趋势总结

### 方案 3：多跳检索（Multi-hop Retrieval）

**核心思路**：搜索分多轮进行，第一轮结果指导第二轮搜索。

```
第一轮: "比较A和B的AI治理方案"
  → 命中文档A的 "AI治理" 片段
  → 从中提取关键词 "隐私保护、算法审计"

第二轮: 用关键词搜索文档B中的相关内容
  → 命中文档B的 "隐私保护" 片段

合并: 将两轮结果组装，提供给 LLM
```

**改动范围**：
- 在 LLM 调用链中加入 "搜索→分析→再搜索" 循环
- 需要 LLM 参与中间决策（判断是否需要补充搜索）
- 修改 `knot-app` 的对话处理流程

**复杂度**：⭐⭐⭐

**收益**：显著改善跨文档推理能力

### 方案 4：文档引用图

**核心思路**：解析文档中的链接（`[text](url)`），构建文档间的引用关系图。

```
文档A: "详见 [报告B](./B.md) 第3章"
         │
         ▼
文档引用图:
  A ──引用──→ B
  B ──引用──→ C
  A ──引用──→ C
```

**改动范围**：
- 在 `knot-markdown` 解析时提取链接目标
- 新增 `DocumentRelation` 存储层
- 搜索时可以沿引用边扩展结果

**复杂度**：⭐⭐⭐

**收益**：自动发现文档间的关联，支持 "相关文档" 推荐

### 方案 5：GraphRAG（知识图谱）

**核心思路**：从文档中提取实体和关系，构建知识图谱，搜索时在图上做推理。

```
文本: "GPT-4 由 OpenAI 开发，使用了 RLHF 训练方法"
         │
         ▼  实体/关系提取
知识图谱:
  [GPT-4] ──开发者──→ [OpenAI]
  [GPT-4] ──训练方法──→ [RLHF]
```

**改动范围**：
- 需要 LLM 进行实体关系提取（Entity-Relation Extraction）
- 新增图数据库存储（或用现有 SQLite 模拟）
- 搜索时既做向量检索，又做图查询
- 完全重构搜索管道

**复杂度**：⭐⭐⭐⭐⭐

**收益**：支持复杂推理（因果、比较、时序），但投入巨大

---

## 五、推荐实施优先级

| 优先级 | 方案           | 收益 | 成本 | 理由                                 |
| ------ | -------------- | ---- | ---- | ------------------------------------ |
| P0     | 上下文窗口扩展 | 高   | 低   | 利用现有 `parent_id`，改动最小       |
| P1     | 文档级摘要索引 | 高   | 低   | 现有 LLM 能力直接可用                |
| P2     | 多跳检索       | 高   | 中   | 显著提升跨文档推理，但需要改对话流程 |
| P3     | 文档引用图     | 中   | 中   | 适合有大量互相引用的文档库           |
| P4     | GraphRAG       | 很高 | 很高 | 长期目标，需要大量基建               |

> **建议**：先实施 P0 + P1，用最小的改动获得最大的提升。P0 只需要改 `KnotStore.search()` 的后处理，P1 只需要在 `KnotIndexer.index_file()` 中加一步摘要生成。
