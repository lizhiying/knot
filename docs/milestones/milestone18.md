# Milestone 18: 结构化数据（Excel/CSV）查询优化与交互重构

## 1. 目标
优化现有基于 RAG 的查询流程，针对 Excel 的 Sheet 和 CSV 等结构化数据，提供基于数据量的智能路由策略、原生的 DuckDB(SQL) 查询分析支持，以及应对多表和复杂条件的交互式多轮查询引导。

## 2. 核心优化需求

### 2.1 结构化数据类型的精准识别与异构处理策略
- **数据类型区分**：检索时需要明确区分结构化数据（Excel Sheet / CSV）与非结构化数据（Text / PDF 等）。
- **基于 Token 容量的智能路由**：
  - **LLM 直连分析（小数据）**：如果目标结构化数据量小，将其转换后连同所有其他必要上下文一起，合计未超过当前 LLM 最大 Token 窗口限制，则直接将数据喂给 LLM，由大模型直接分析回答。
  - **SQL 查询引擎（大数据，依托 DuckDB）**：如果数据量较大，必须使用 DuckDB。系统需自动将结构化数据转入 DuckDB，使用 LLM 将用户意图转化为多步 SQL 查询，并在前端直接渲染展示查询出的结果数据表。
- **多步拆解与引导确认机制**：当用户的查询过于复杂，无法一次性通过一条 SQL 获取准确数据，或者存在条件歧义时，系统不应强行生成并报错。而是通过多次对话，一步步引导用户补充条件，或对中间数据进行确认筛选。

### 2.2 命中多表时的强制用户降噪选择
- **场景**：用户的自然语言查询导致向量或关键字引擎检索出了**多个**被认为相关的表格数据（例如多个同类 Sheet 或多个年份的 CSV）。
- **策略**：此时系统不应尝试处理所有表使其超时或上下文爆炸，**必须先中断自动查询流程**。
- **交互**：系统应在界面抛出候选表格列表，让用户手动选择明确要查询的目标表。用户选定后再基于该特定表继续后续的数据分析与响应。

### 2.3 混合检索结果的 LLM 智能预过滤
- **场景**：检索模块可能由于词条的高相似性，同时拉取出结构化数据（Excel / CSV）及其他类型的文档（如 PDF 文档、帮助手册 TXT 等）。
- **策略**：在将所有物料混合送入最终回答环节（或分析环节）前，插入一个前置的 **LLM 预判定层**。
- **操作**：请求 LLM 快速判断检索出的这些“非结构化文件”与用户当下的提问是否真实存在语义和逻辑上的相关性。
- **结果**：过滤掉那些仅仅是关键字命中但牛头不对马嘴的杂音文档，仅保留确认为强相关的上下文，随后再进入核心业务处理流。这能大幅提升最终准确率并节约 Token。

## 3. 实现路径规划 (Task Breakdown)

### 任务 1：核心 RAG 检索管线的改造 —— 混合数据预过滤
- [ ] 拆分检索结果链路，识别数据类型（结构化数组 vs 文本 Chunk 组）。
- [ ] 针对文本 Chunk 组构建前置评估 Prompt，让 LLM 以最短时间判断其相关性。
- [ ] 将过滤后保留的高质量相关文本作为最终补充上下文。

### 任务 2：多表检出交互与状态保存
- [ ] 改写 Router 逻辑：如果返回结果类型标记为 `Structured_Data_List` 且 `count > 1`，则进入等待状态（Pending）。
- [ ] 前端提供针对表格的选择器 UI 组件（供用户打钩选择）。
- [ ] 用户选择后，将指定表的唯一标识或描述回传后端，闭环后接续查询动作。

### 任务 3：动态 Token 预算与引擎路由
- [ ] 实现针对表格数据的快速 Token 估算器（Token Estimator）。
- [ ] 根据估算结果和当前上下文占用的动态预算，自动选择走 `Direct LLM Q&A` 还是走 `SQL Engine` 逻辑分支。

### 任务 4：基于 DuckDB 的多步执行框架与数据回显
- [ ] 完善后台 DuckDB 的上下文加载机制。
- [ ] 引入 `Agentic Step-by-Step` 工作流理念：LLM 编写 SQL -> 执行出错或条件不足 -> 生成向用户提问的 Prompt；执行成功 -> 判断是否为最终目标 -> 若是则将数据透传到前端 Grid 渲染，若需继续聚合则接力下一步。
