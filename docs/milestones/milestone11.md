# Milestone 11: 分词深度优化与多语言搜索增强

## 1. 目标
- **保持优势**：中文继续由 Jieba 负责，确保符合语感的切分。
- **增强英文**：引入 Stemmer（词干提取）和 Lowercase，解决 run/running 匹配问题。
- **泛语言兜底**：引入 ICU Tokenizer，解决德法西俄、泰越等小众语言及奇怪符号的召回。
- **解决混写**：通过“多字段复制存储”，确保中英混写、中意混写都能在对应字段命。
- **降噪过滤**：解决如 "s s s 入门" 导致命中大量无效单个字母 "s" 的问题。

---

## 2. 字段与分词配置 (Schema Upgrade)

| 字段                | 分词器 (Tokenizer) | 权重 (Boost) | 说明                                                |
| :------------------ | :----------------- | :----------- | :-------------------------------------------------- |
| **`text_zh`**       | `jieba`            | **5.0**      | 中文核心字段，负责精准中文匹配。                    |
| **`text_std`**      | `en_knot`          | **3.0**      | 英文主力，包含 Lowercase + Stemmer + Stopwords。    |
| **`text_icu`**      | `icu`              | **1.0**      | **新增**。泛语言兜底，Unicode 全兼容切分。          |
| **`file_name`**     | `jieba`            | **8.0**      | 文件名中文部分。                                    |
| **`file_name_std`** | `default`          | **5.0**      | **新增**。文件名英文部分，防止 Jieba 切坏英文单词。 |
| **`path_tags`**     | `default`          | **2.0**      | 结构化标签。                                        |

---

## 3. 索引写入策略 (Indexing Strategy)

采用 **"空间换确定性"** 的策略。在写入索引时，同一份正文内容会分发到不同字段：

1. **正文内容 (`text`)**：
   - 总是写入 `text_std`。
   - 总是写入 `text_icu`。
   - **正则判断**：如果包含 CJK 字符，则写入 `text_zh`。
2. **文件名 (`file_name`)**：
   - 总是写入 `file_name_std`。
   - 如果包含 CJK 字符，则写入 `file_name` (Jieba)。

---

## 4. 查询策略 (Query Strategy)

### 多字段加权 (Multi-field Boost)
QueryParser 将同时检索多个字段，利用权重区分优先级。
- 如果用户搜中文，`text_zh` 权重最高，结果排在前面。
- 如果搜中英混写，各字段分别命中，分数叠加。
- `text_icu` 只在主字段都没活干时出来“兜底”，不会干扰正常排名。

### 无效结果过滤 (Noise Reduction)
针对用户反馈的 "s s s 入门" 干扰问题：
1. **长度惩罚**：对长度小于 2 的非中文字符增加惩罚分，或将其视为 Stopword。
2. **匹配密度检查**：如果结果仅命中极其简短且高频的单个字母（如 's'），且没有命中查询中的核心词（如“入门”），则将其从 Hybrid 结果中剔除。
3. **分词预处理**：继续保持 `preprocess_query` 的中英边界插空逻辑，辅助分词。

## 5. 实施路线图

### 第一阶段：基础设施 (Rust 层)
1. 添加 `tantivy-analysis-contrib` 依赖（包含 ICU 支持）。
2. 在 `store.rs` 中定义 `en_knot` (Simple + Lowercase + Stemmer + Stopwords)。
3. 注册 `icu` 分词器。

### 第二阶段：Schema 迁移
1. 修改 `create_fts_index` 增加 `text_icu` 和 `file_name_std` 字段。
2. 执行索引版本检测，如果 Schema 变化则触发全文重建（Reindex）。

### 第三阶段：写入与查询优化
1. 修改 `add_document` 逻辑，实现多字段分发。
2. 修改 `search` 方法，更新 `QueryParser` 的字段列表与权重设置。
3. 增加“无效结果”过滤逻辑。

---

## 6. 成功指标
1. **中英混写**：搜索 "JSON解析" 能够同时利用 `text_zh` 和 `text_std` 的优势。
2. **英文泛化**：搜 "models" 必须能命中包含 "model" 的文档。
3. **噪音控制**：搜 "a s d s 入门"，结果列表前几名必须是包含“入门”的，而非包含大量 "s" 的无效文档。
4. **多语言**：俄语、日语等纯外语文档在不做特殊配置下能被 `text_icu` 检索到。
